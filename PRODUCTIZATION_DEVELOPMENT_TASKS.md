# 密码攻击风险检测与隐私训练平台产品化开发任务文档

## 1. 产品定位

本项目后续不再继续堆叠概念功能，而是围绕一条实用主线推进：

> 面向平台方的密码攻击风险检测与隐私训练系统。

系统分为两个角色：

- 用户端：上传登录安全数据，获得风险检测、原因解释、处理建议和分析报告。
- 管理端：管理加密归档数据、审核训练数据、执行模型训练和联邦训练、查看系统审计。

技术主线必须保持清晰：

```text
用户数据上传
→ 数据体检与脱敏
→ 风险检测与原因解释
→ 报告生成
→ AES 加密归档
→ 管理端审核
→ 本地训练 / 四节点联邦训练
→ 模型版本管理
→ 系统审计
```

联邦学习、Paillier、自适应优化、安全审计都应服务这条主线，不能独立堆功能。

## 2. 当前主要问题

### 2.1 产品定位问题

当前系统已经有用户端和管理端雏形，但部分页面仍像实验展示：

- 用户端风险检测和报告可以跑通，但原因解释、建议、报告格式还不够产品化。
- 管理端能训练和看审计，但数据生命周期、模型版本和指标可信度不足。
- 联邦学习、Paillier、自适应优化的价值需要在管理端明确解释和可视化。

### 2.2 隐私可信度问题

系统声称隐私保护，但必须进一步确保：

- 用户名、手机号、密码、银行卡等敏感字段不在前端、报告、日志中明文暴露。
- 管理端默认只展示摘要、脱敏预览和训练状态。
- 密码字段不进入训练，只转化为长度、强度、复杂度等派生特征。
- AES 用于文件加密归档，Paillier 用于密态字段展示和安全聚合方向说明，不能写成完整密文训练平台。

### 2.3 训练可信度问题

管理端训练结果如果直接显示 `1.0000`，容易被质疑：

- 是否训练集和测试集混用。
- 是否仅使用生成数据。
- 是否过拟合。
- 是否是演示指标。

训练页面必须标注指标来源，并展示训练数据来源、样本量、标签分布、模型版本和是否启用。

### 2.4 安全与工程可信度问题

公网展示时需要补齐基础安全边界：

- 管理端登录配置必须明确。
- 上传文件大小、类型、行列数需要限制。
- 日志和报告不能泄露敏感字段。
- CORS、session、secret key 等部署项需要说明。
- 系统审计需要从日志列表升级为可筛选、可检索、可解释的审计工作台。

## 3. 开发原则

1. 每次只做一个小任务。
2. 每个任务必须能单独验收。
3. 不改 Flask / SQLite 技术栈。
4. 不引入 React / Vue / Angular。
5. 不引入 Redis / Celery / 大型服务。
6. 不删除现有可用接口。
7. 不把实验原型写成生产级平台。
8. 用户端优先保证上传、检测、解释、报告闭环。
9. 管理端优先保证数据治理、训练、模型版本、审计闭环。
10. 所有敏感字段展示前必须脱敏或转派生特征。

## 4. 任务路线图

## Task P1：用户端敏感字段统一脱敏

### 目标

确保用户端、管理端、报告和日志中不直接展示敏感字段明文。

### 允许修改

- `src/user_submission_manager.py`
- `app.py` 中用户提交相关接口的最小适配
- `index.html`
- 必要测试文件

### 禁止修改

- 联邦训练核心算法
- 攻击检测核心模型
- 数据库技术栈
- 管理端登录结构

### 实现要点

- 统一识别字段：`password`、`phone`、`mobile`、`bank_card`、`id_card`、`email`、`username`。
- 密码不展示、不入库明文、不进入训练。
- 手机号、银行卡、邮箱、用户名统一脱敏。
- 密码只生成派生特征：
  - 长度
  - 是否包含数字
  - 是否包含大小写
  - 是否包含特殊字符
  - 强度等级
- 报告中只出现脱敏值和派生特征。

### 验收标准

- 上传包含 `password` 的 CSV 后，前端和报告看不到密码明文。
- 上传包含手机号、邮箱、用户名后，页面展示脱敏值。
- 风险检测仍可使用派生特征。
- 管理端提交详情不显示原始敏感字段。

### 回归命令

```bash
python -m py_compile app.py src/**/*.py
python app.py
```

## Task P2：上传数据 Schema 检查与字段识别

### 目标

让系统能判断用户上传的数据是否适合风险检测，并给出明确提示。

### 允许修改

- `src/user_submission_manager.py`
- `index.html`
- 必要测试文件

### 实现要点

- 识别登录安全字段：
  - `username`
  - `ip`
  - `device`
  - `browser`
  - `os`
  - `login_success`
  - `failed_attempts`
  - `request_rate`
  - `response_time_ms`
  - `session_duration`
  - `timestamp`
  - `label`
- 输出字段完整度评分。
- 输出缺失字段清单。
- 输出是否适合直接检测。
- 字段不足时允许上传，但提示检测可信度较低。

### 验收标准

- 上传完整登录安全样本时显示“字段较完整”。
- 上传普通隐私样本时提示“不适合直接做攻击风险检测”。
- 页面不崩溃，报告中保留字段体检结论。

## Task P3：风险检测 Top 20 样本详情与解释页

### 目标

让用户知道每条高风险结果为什么高，而不是只看到风险分数。

### 允许修改

- `src/user_submission_manager.py`
- `index.html`

### 实现要点

- 风险检测结果按风险分数排序。
- 展示 Top 20 高风险样本。
- 每条样本可展开查看：
  - 风险分数构成
  - 触发特征
  - 字段值
  - 阈值
  - 中文原因
  - 处理建议
- 原因必须来自登录安全字段，例如：
  - 失败次数高
  - 请求频率异常
  - 响应时间异常
  - 异常设备
  - 异常时间段
  - 标签与模型判断不一致

### 验收标准

- 每条高风险样本至少有一个明确原因。
- 原因里包含字段名、当前值、判断依据。
- 不再出现“手机号/收入/信用分导致攻击风险”的逻辑错误。

### 允许修改

- 用户报告接口的最小适配

### 实现要点

  - 样本数
  - 高风险数
  - 中风险数

  - 风险摘要

## Task P5：管理端数据生命周期

### 目标

让管理端数据池像真实数据治理流程，而不是简单列表。

### 允许修改

- `app.py`
- `src/user_submission_manager.py`
- SQLite 初始化相关代码
- `index.html`

### 实现要点

每条提交数据必须有状态：

```text
待审核
→ 已加密归档
→ 可训练
→ 已用于训练
→ 已生成模型版本
```

管理端显示：

- 上传时间
- 文件名
- 样本数
- 字段数
- 标签列
- 风险摘要
- 加密归档状态
- 当前生命周期状态
- 最近操作时间

### 验收标准

- 管理员能清楚看到每份数据当前处于哪个阶段。
- 标记可训练后，训练中心能选择该数据。
- 数据用于训练后状态更新。

## Task P6：训练指标可信度标注

### 目标

避免训练结果看起来像伪造或过拟合。

### 允许修改

### 实现要点

### 验收标准

- 页面不再只显示一个 `1.0000`。
- 用户能看出指标来自哪里。
- 联邦训练曲线变化平滑、可解释。

## 5. 推荐执行顺序

建议按下面顺序交给 Codex 追求目标模式：

```text
P1 用户端敏感字段统一脱敏
P2 上传数据 Schema 检查与字段识别
P3 风险检测 Top 20 样本详情与解释页
P4 分析报告页面产品化
P5 管理端数据生命周期
P6 训练指标可信度标注
P7 模型版本管理页面
P8 系统审计筛选、分页和详情
P9 上传安全边界强化
P10 部署安全配置清单
```

原因：

- 先解决隐私可信度。
- 再解决用户端检测结果是否可信。
- 再解决管理端数据治理和训练可信度。
- 最后补齐审计和部署安全。

## 7. 当前阶段不建议做的事

暂时不要做：

- 新增更多算法名词。
- 新增复杂账号体系。
- 新增分布式服务。
- 新增第三方大模型解释。
- 新增完整 SOC / WAF 功能。
- 把 Paillier 写成完整密文训练。
- 把联邦训练写成真实跨机构生产部署。

当前最重要的是把已有链路做深、做稳、做可信。
