# 密码攻击检测与隐私训练平台实用性架构审计

审计时间：2026-06-18  
审计范围：当前 `master` 工作区代码，重点查看 `app.py`、`index.html`、`src/user_submission_manager.py`、`src/utils/data_storage.py`、`src/security/*`、`src/detection/*`、`src/federated/*`。

## 1. 总体结论

当前项目已经从早期“答辩展示页”推进为一个轻量双端平台雏形：

- 用户端：面向数据上传、敏感数据保护、风险检测、风险解释和报告留存。
- 管理端：面向用户提交数据池、管理员登录、本地训练、四节点模拟联邦训练、模型版本和训练任务记录。
- 系统审计：面向访问记录、安全事件、TraceId、慢接口和限流事件展示。

项目方向与“密码攻击检测与隐私训练平台”基本一致，但当前仍是轻量实验原型，不应描述为生产级平台。主要短板集中在：敏感字段脱敏规则不够系统、风险摘要展示仍有 JSON 化痕迹、训练指标容易误导、模型版本启用流程不完整、系统审计缺少筛选分页和详情展开。

当前最应该继续做的是补齐实用性闭环，而不是继续增加大模块。

## 2. 当前功能清单

### 2.1 用户端

入口：`index.html` 根据端口判断用户端模式，5000 端口显示用户端导航。

当前用户端主要页面：

- 上传数据
- 风险检测
- 分析报告
- 风险分数解释页（隐藏页面，通过检测结果中的“分数解释”进入）

当前用户端相关接口：

- `POST /api/user/datasets/upload`
- `POST /api/user/datasets/<submission_id>/analyze`
- `GET /api/user/reports/<submission_id>`
- `POST /api/generate_login_security_dataset`
- `POST /api/generate_privacy_dataset`

当前已具备的能力：

- 支持 CSV/JSON 上传。
- 上传后由 `UserSubmissionManager.create_submission()` 建立提交记录。
- 原始文件会进入 AES 加密归档流程。
- 上传预览使用 `masked_preview`，不会直接把全部原始数据暴露给前端。
- `password` 等字段会转换为派生特征，例如密码长度、强度等。
- 风险分析通过 `UserSubmissionManager.analyze()` 调用检测器并生成风险摘要。
- 风险检测结果保留 Top 20 中高风险样本解释入口。
- 报告通过 Markdown 文件生成并由 `GET /api/user/reports/<submission_id>` 读取。
- 用户端提供“登录安全样本”和“隐私加密样本”两类生成入口。

当前不足：

- 敏感字段脱敏没有形成统一、可复用、可审计的字段规则表。
- 上传预览仍需要确认手机号、token、身份证号、银行卡号、薪资等字段不会以明文显示。
- 报告页仍偏文本，不够结构化，缺少摘要卡片。
- 用户端没有真正的用户账号体系，本次架构应继续保持轻量，不建议现在加入完整多用户认证。

### 2.2 管理端

入口：`index.html` 根据端口判断管理端模式，5001 端口显示管理端导航。

当前管理端主要页面：

- 数据管理
- 训练中心
- 系统审计
- 管理员登录页

当前管理端相关接口：

- `POST /api/admin/login`
- `POST /api/admin/logout`
- `GET /api/admin/session`
- `GET /api/admin/submissions`
- `GET /api/admin/submissions/<submission_id>`
- `POST /api/admin/submissions/<submission_id>/archive`
- `POST /api/admin/submissions/<submission_id>/mark-trainable`
- `POST /api/admin/training/local`
- `POST /api/admin/training/federated`
- `GET /api/admin/training/tasks`
- `GET /api/admin/model-versions`
- `GET /api/admin/audit/events`

当前已具备的能力：

- 管理端有最小管理员登录，基于 Flask session。
- 管理 API 有基础登录保护。
- 管理端可以查看用户提交列表。
- 可将用户提交标记为可训练。
- 本地训练接口会读取可训练提交数据并调用现有检测模型训练。
- 联邦训练接口会将可训练数据划分为四个节点，执行本地四节点 FedAvg 流程。
- 训练任务和模型版本会写入 SQLite 的 `training_tasks`、`model_versions`。
- 管理端保留系统已有数据集状态和联邦节点状态查看。

当前不足：

- 管理端默认账号密码如果未配置环境变量，存在默认值风险。
- 数据管理页风险摘要仍容易显示为 JSON 字符串，不适合实际管理使用。
- 标记可训练流程过于简单，缺少“待审核 / 已归档 / 可训练 / 已拒绝”的清晰状态流转展示。
- 本地训练和联邦训练的指标如果出现 1.0000，必须明确标注为训练集指标或原型指标。
- 模型版本只有记录，没有“当前启用版本”切换和可视化对比。

### 2.3 系统审计

当前系统审计相关实现：

- `src/security/middleware.py`
- `src/security/security_logger.py`
- `src/security/events_api.py`
- `src/security/ip_geo.py`
- `src/security/user_agent_parser.py`
- `GET /api/security/events/recent`
- `GET /api/admin/audit/events`

当前已具备的能力：

- TraceId 请求链路追踪。
- 慢接口日志。
- 限流触发事件。
- `site_visit` 页面访问记录。
- 最近访问记录中包含 IP、User-Agent、设备、浏览器、系统、端口、路径、TraceId 等信息。
- IP 归属地采用本地离线库优先、无库时显示 local/private/unknown 的策略。

当前不足：

- 审计页面还缺少筛选、分页、长字段截断和详情展开。
- 安全事件与访问记录需要继续区分：安全事件用于说明异常和处理建议，访问记录用于说明谁访问过平台。
- 不应扩大成 SOC/SIEM 或 WAF 平台。

## 3. 真实可用与模拟边界

### 3.1 真实可用

以下能力在当前代码中已经有真实路径和接口支撑：

- Flask 双端页面分流：用户端 5000、管理端 5001。
- CSV/JSON 文件上传。
- AES 加密归档用户上传文件。
- 用户提交索引和 SQLite 记录。
- 密码字段派生特征处理。
- 风险检测接口调用和风险摘要返回。
- Top 20 中高风险样本解释数据结构。
- Markdown 报告生成与下载。
- 管理员最小登录和管理 API 保护。
- 可训练数据池标记。
- 本地轻量模型训练接口。
- 四节点本地联邦训练流程。
- 训练任务和模型版本 SQLite 记录。
- TraceId、慢接口、限流、访问记录、安全事件日志。

### 3.2 半完成或轻量原型

以下能力有代码路径，但还不能按生产级能力宣传：

- 联邦学习：当前是本地四节点模拟训练，不是真实多机构、多进程、多节点通信。
- Paillier：当前适合展示密态字段和安全聚合概念，不是完整密文训练。
- 模型训练：可运行，但评估指标需要明确训练集/验证集来源，不能只展示 accuracy。
- 模型版本：有记录表和接口，但缺少当前启用版本、回滚、对比和切换。
- 系统审计：有事件和访问记录，但缺少实用管理筛选能力。
- 管理端权限：有单管理员 session，但不是完整账号、权限、审计体系。

### 3.3 当前不应做或不能做

以下方向不符合项目定位或服务器限制：

- 不要做 WAF。
- 不要做漏洞扫描器。
- 不要做区块链账本或 IPFS。
- 不要做 SOC/SIEM。
- 不要引入 TensorFlow / PyTorch。
- 不要接入 Kubernetes / FATE 真实集群。
- 不要引入大模型解释风险。
- 不要把系统包装为生产级银行/医院安全平台。

## 4. 当前最影响实用性的 10 个问题

1. **敏感字段脱敏不够系统**  
   当前有脱敏和密码派生处理，但需要统一字段规则。手机号、token、身份证、银行卡、邮箱、薪资、地址、密码等字段都应有明确策略。

2. **上传预览仍需安全审查**  
   用户上传的数据预览必须默认展示脱敏字段，不应出现任何明文密码、token、完整手机号或完整银行卡号。

3. **风险检测结果解释还不够稳定**  
   已有 Top 20 和分数解释入口，但解释内容需要进一步与具体登录安全字段绑定，避免泛泛而谈。

4. **报告页信息组织不够实用**  
   当前报告以 Markdown 为主，应增加摘要卡片、风险分布、Top 高风险样本、建议列表和隐私保护说明。

5. **管理端风险摘要展示不友好**  
   数据管理页不应显示 JSON 字符串，应改为高/中/低风险标签和样本数卡片。

6. **训练指标容易误导**  
   如果 accuracy 出现 1.0000，必须标注“训练集指标 / 原型指标 / 非独立验证集指标”，并逐步加入验证集或 holdout 指标。

7. **模型版本记录缺少启用状态**  
   当前有版本记录，但缺少“当前启用版本”、启用按钮、版本对比、训练来源说明。

8. **用户提交数据进入训练池的授权链路过短**  
   标记可训练目前过于直接，应明确管理员确认数据质量、授权范围、隐私处理状态后再进入训练。

9. **系统审计缺少实用检索能力**  
   需要筛选风险级别、事件类型、IP、路径、时间范围，并对长字段进行截断和详情展开。

10. **数据来源和用途说明仍需统一**  
    需要清楚区分登录安全样本、隐私加密样本、UNSW/本地生成训练集、用户上传数据，避免“隐私数据导致攻击风险”的逻辑混乱。

## 6. 下一步 10 个小任务

### P0-1 用户端敏感字段脱敏预览

目标：建立统一脱敏工具和字段规则，确保上传预览、管理端脱敏预览、报告样本摘要都不泄露敏感字段。  
建议修改：`src/user_submission_manager.py`、`index.html`。  
验收标准：

- `password` 不显示明文，只显示长度、强度、是否弱密码。
- 手机号、邮箱、token、身份证、银行卡、薪资等字段默认脱敏。
- 前端表格内容全部 HTML 转义。

### P0-2 风险检测 Top 20 + 风险解释入口优化

目标：让风险检测页默认展示 Top 20 中高风险样本，并提供清晰的分数解释页面。  
建议修改：`index.html`、`src/user_submission_manager.py`。  
验收标准：

- Top 20 按风险分数降序。
- 每条包含触发特征、中文原因、处理建议。
- 分数解释说明模型信号和规则信号，不夸大为绝对结论。

### P0-3 管理端风险摘要 JSON 改成标签/卡片

目标：数据管理页不再显示 JSON 字符串。  
建议修改：`index.html`。  
验收标准：

- 显示高风险、中风险、低风险、总样本标签。
- 无数据时显示空状态。
- 不删除原接口。

### P1-4 报告页结构化展示

目标：分析报告页增加摘要卡片、风险分布、高风险样本、建议和隐私说明。  
建议修改：`index.html`。  
验收标准：

- Markdown 报告保留。
- 页面上方有可读摘要卡片。
- 下载 Markdown 按钮保留。

### P1-5 训练指标标注与验证集提示

目标：避免训练精度 1.0000 误导。  
建议修改：`app.py`、`index.html`。  
验收标准：

- 本地训练和联邦训练指标明确标注“训练集指标”或“验证集指标”。
- 若没有独立验证集，前端明确提示。

### P1-6 模型版本当前启用状态

目标：补齐模型版本基础管理。  
建议修改：`src/utils/data_storage.py`、`app.py`、`index.html`。  
验收标准：

- 模型版本列表显示版本号、来源、样本数、算法、指标、时间、是否当前启用。
- 暂不需要 MLflow。

### P1-7 用户提交训练授权流转

目标：让用户提交从待审核到可训练有清晰状态。  
建议修改：`src/user_submission_manager.py`、`app.py`、`index.html`。  
验收标准：

- 支持待审核、已归档、可训练、已拒绝。
- 管理端可查看状态说明。

### P2-8 系统审计筛选和分页

目标：保留现有审计页，但提升可用性。  
建议修改：`index.html`、必要时扩展 `/api/admin/audit/events` 查询参数。  
验收标准：

- 支持事件类型、风险等级、IP、路径筛选。
- 支持分页或限制条数。
- 长字段截断并可展开详情。

### P2-9 管理端登录安全加固

目标：保持轻量，但避免默认弱配置。  
建议修改：`app.py`、`config` 或 `.env` 说明文档。  
验收标准：

- 生产环境必须从环境变量读取管理员密码。
- 默认密码只允许本地开发提示，不建议线上使用。
- 登录失败返回统一错误，不泄露具体原因。

### P3-10 轻量回归脚本

目标：每次修改后快速验证 5000/5001 关键链路。  
建议新增：`scripts/smoke_check.py` 或文档化 curl 命令。  
验收标准：

- 检查用户端首页、管理端首页、健康检查、上传接口、审计接口。
- 不依赖重型测试框架。

## 7. 推荐实施顺序

建议先做：

1. 用户端敏感字段脱敏预览。
2. 风险检测结果 Top 20 + 风险解释入口优化。
3. 管理端风险摘要 JSON 改成标签/卡片。

理由：

- 这三项直接影响用户是否信任系统。
- 不需要引入新依赖。
- 不需要改数据库结构或大规模重构。
- 能明显提升实用性和可视化效果。

## 8. 当前边界说明

当前系统可以作为轻量原型继续推进，但必须坚持以下边界：

- 用户上传数据可以用于风险检测和加密归档，但不能宣称完整合规数据治理。
- 管理端训练可以使用管理员确认的数据，但不能宣称真实多机构生产训练。
- Paillier 可以用于密态字段展示和安全聚合方向说明，但不能宣称完整密文机器学习。
- 系统审计用于平台访问可观测，不是 SOC/SIEM。
- 所有页面必须明确区分真实接口返回、原型指标、训练集指标和模拟流程。
